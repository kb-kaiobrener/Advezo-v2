## Story

**As a** gestor de tráfego,
**I want to** receber alerta via WhatsApp quando o saldo de uma conta estiver projetado para esgotar em menos de N dias,
**so that** eu possa agir antes da campanha ser pausada por saldo insuficiente.

## Acceptance Criteria

- [x] **AC 3.6.1** — Cron route `GET /api/cron/send-alerts` (Vercel Cron, a cada 15 min `*/15 * * * *`): protegida por `Authorization: Bearer ${CRON_SECRET}` (401 se ausente ou inválido). Busca alertas com `resolved_at IS NULL AND whatsapp_sent_at IS NULL`. Para cada alerta, obtém a conta WhatsApp conectada do workspace (`whatsapp_accounts` com `status = 'connected'`) e o destino configurado (`alert_destination_id`).

- [x] **AC 3.6.2** — Dedup via claim atômico: antes de chamar o worker, executa `UPDATE alerts SET whatsapp_sent_at = NOW(), whatsapp_destination_id = $destination WHERE id = $id AND whatsapp_sent_at IS NULL RETURNING id`. Se `RETURNING` retornar vazio (outro processo já enviou), skip silencioso. Garante que o mesmo alerta nunca é enviado duas vezes, mesmo em cenário de retry ou reinício do cron.

- [x] **AC 3.6.3** — Falha no worker: executa `UPDATE alerts SET whatsapp_sent_at = NULL, whatsapp_last_error = $erro WHERE id = $id` (desfaz o claim). Alerta fica com `whatsapp_sent_at IS NULL` e será retentado no próximo ciclo do cron. `whatsapp_last_error` preserva o último erro para diagnóstico.

- [x] **AC 3.6.4** — Suporte a grupo: destinatário pode ser JID `@g.us`. O worker já suporta grupos após o fix da Story 3.5 (T1). A validação de formato é idêntica à das Stories 3.3 e 3.5: `/^\d+@g\.us$/` para grupos, `/^\d{10,15}$/` para individuais.

- [x] **AC 3.6.5** — Alertas sem destino configurado (`whatsapp_accounts.alert_destination_id IS NULL`) são ignorados silenciosamente pelo cron (sem log de erro — é estado esperado para contas não configuradas).

### Cron Route — send-alerts (`apps/web/src/app/api/cron/send-alerts/route.ts`)

```typescript
import { NextRequest, NextResponse } from 'next/server'
import { createSupabaseServiceClient } from '@/lib/supabase/server'

export async function GET(req: NextRequest) {
  if (req.headers.get('Authorization') !== `Bearer ${process.env.CRON_SECRET}`) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }

  const supabase = createSupabaseServiceClient()

  // 1. Alertas ativos não enviados — join com ad_accounts é válido (FK real
  //    alerts.ad_account_id → ad_accounts.id). NÃO embutir whatsapp_accounts:
  //    não existe FK entre alerts e whatsapp_accounts, o embed falharia.
  const { data: alerts } = await supabase
    .from('alerts')
    .select(`
      id, workspace_id, ad_account_id, alert_type, threshold_days, projected_days,
      ad_accounts!inner(name)
    `)
    .is('resolved_at', null)
    .is('whatsapp_sent_at', null)

  if (!alerts?.length) return NextResponse.json({ processed: 0 })

  // 2. Contas WhatsApp conectadas com destino de alerta configurado, indexadas
  //    por workspace_id — cruzamento feito no código, não no PostgREST.
  const workspaceIds = [...new Set(alerts.map((a) => a.workspace_id))]
  const { data: waAccounts } = await supabase
    .from('whatsapp_accounts')
    .select('workspace_id, account_id, alert_destination_type, alert_destination_id')
    .in('workspace_id', workspaceIds)
    .eq('status', 'connected')
    .not('alert_destination_id', 'is', null)

  const waByWorkspace = new Map<string, { account_id: string; alert_destination_id: string }>()
  for (const wa of waAccounts ?? []) {
    if (!waByWorkspace.has(wa.workspace_id)) waByWorkspace.set(wa.workspace_id, wa)
  }

  let sent = 0, skipped = 0, failed = 0

  for (const alert of alerts) {
    const wa = waByWorkspace.get(alert.workspace_id)
    const adAccount = Array.isArray(alert.ad_accounts) ? alert.ad_accounts[0] : alert.ad_accounts
    if (!wa) { skipped++; continue }  // sem conta conectada ou sem destino configurado (AC 3.6.5)

    // 3. Claim atômico — tenta marcar como enviado antes de chamar o worker
    const { data: claimed } = await supabase
      .from('alerts')
      .update({ whatsapp_sent_at: new Date().toISOString(), whatsapp_destination_id: wa.alert_destination_id })
      .eq('id', alert.id)
      .is('whatsapp_sent_at', null)
      .select('id')
      .single()

    if (!claimed) { skipped++; continue }  // já enviado por outro processo

    // 4. Montar mensagem
    const projectedDate = new Date()
    projectedDate.setDate(projectedDate.getDate() + Math.round(alert.projected_days))
    const dataFormatada = projectedDate.toLocaleDateString('pt-BR')

    const text = [
      `⚠️ Alerta de Saldo — ${adAccount.name}`,
      `Saldo projetado para esgotar em ${Math.round(alert.projected_days)} dias.`,
      `Limite configurado: ${alert.threshold_days} dias de veiculação.`,
      `Ação sugerida: recarregue o saldo da conta antes de ${dataFormatada}.`,
    ].join('\n')

    // 5. Enviar via worker
    const workerRes = await fetch(`${process.env.WHATSAPP_WORKER_URL}/send`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        workspace_id: alert.workspace_id,
        account_id: wa.account_id,
        to: wa.alert_destination_id,
        text,
      }),
    }).catch(() => null)

    if (workerRes?.ok) {
      // whatsapp_sent_at já está setado pelo claim — limpar eventual last_error
      await supabase.from('alerts').update({ whatsapp_last_error: null }).eq('id', alert.id)
      sent++
    } else {
      // 6. Rollback do claim — alerta voltará a ser tentado no próximo cron
      const err = workerRes ? (await workerRes.text().catch(() => 'erro desconhecido')) : 'worker inacessível'
      await supabase
        .from('alerts')
        .update({ whatsapp_sent_at: null, whatsapp_last_error: err })
        .eq('id', alert.id)
      failed++
    }
  }

  return NextResponse.json({ processed: alerts.length, sent, skipped, failed })
}
```

**Garantia de dedup:** O UPDATE com `.is('whatsapp_sent_at', null)` é atômico no Postgres. Apenas um dos processos concorrentes receberá um registro de volta em `.select('id').single()` — o outro não terá RETURNING e será descartado como `skipped`.

**Crash após claim antes de rollback:** Se o cron morrer após `whatsapp_sent_at` ser setado (claim) mas antes do worker call ou do rollback, o alerta ficará com `whatsapp_sent_at IS NOT NULL` sem ter sido realmente enviado. Esse edge case é aceito — preferência pelo no-dup sobre guaranteed delivery, conforme requisito.

O JOIN `ad_accounts!inner(name)` requer que a `alert` tenha um `ad_account_id` válido (garantido pelo FK). O cron usa service role — ignora RLS. O campo `name` é necessário para montar a mensagem (AC 3.6.8).

#### Verdicto: **FAIL**

O desenho de dedup (claim atômico + rollback) está conceitualmente correto e bem coberto por teste unitário. **Porém, o teste manual das rotas (REST direto com service key real) expôs dois bloqueadores que tornam o cron `send-alerts` completamente não-funcional em produção.** Ambos passaram despercebidos porque os testes unitários usam mock do banco, que não reflete o schema nem as permissões reais.

#### 🔴 Bloqueadores (must-fix)

- **BLOCK-001 (high, defeito de código desta story):** O cron `send-alerts/route.ts` L29 usa `ad_accounts!inner(name)` e L88 lê `adAccount.name`. **A coluna não existe** — em `ad_accounts` (migration 000004) o campo é `account_name`. Verificado via REST: a query retorna `400 — column ad_accounts_1.name does not exist`. Consequência: a primeira consulta do cron falha, `alertsError` dispara e a rota retorna 500 — **nenhum alerta é jamais enviado**. O teste `alert-destination.test.ts` passou porque o mock devolvia `ad_accounts: { name: '...' }`, mascarando o erro.
  - **Fix:** trocar `ad_accounts!inner(name)` → `ad_accounts!inner(account_name)` e `adAccount.name` → `adAccount.account_name` na rota; ajustar o mock do teste para `{ account_name: '...' }` (o mock atual valida a coluna errada).

- **BLOCK-002 (high, lacuna de GRANT — bloqueia este cron):** O cron lê e faz **claim (UPDATE)** em `alerts` via `service_role`. A tabela `alerts` (migration 000008) **não tem GRANT para service_role** — REST retorna `403 permission denied`, hint `GRANT SELECT ON public.alerts TO service_role`. Sem SELECT **e UPDATE**, nem a leitura inicial nem o claim atômico (AC 3.6.2) nem o rollback (AC 3.6.3) funcionam. `ad_accounts` (embed) tem o mesmo 403.
  - **Fix:** migration de grants (precedente: 000011 `whatsapp_accounts_grants`) com `GRANT SELECT, INSERT, UPDATE, DELETE ON public.alerts TO service_role` e `GRANT SELECT ON public.ad_accounts TO service_role` (+ `ad_campaigns`, `campaign_metrics` para desbloquear a 3.5 no mesmo migration).
