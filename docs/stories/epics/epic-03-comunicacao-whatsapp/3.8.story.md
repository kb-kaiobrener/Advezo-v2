## Story

**As a** cliente final de uma agência,
**I want to** acessar um painel com login próprio para ver os dados das minhas contas de anúncio,
**so that** eu acompanhe meus resultados sem depender de prints ou planilhas — e sem jamais ver dados de outros clientes.

## Acceptance Criteria

- [ ] **AC 3.8.1** — Fluxo de convite: na página `/clients/[id]/configuracoes`, seção "Acesso do Cliente" com campo de email e botão "Enviar convite". Server Action `inviteClientUser(clientId, email)` (somente gestor autenticado do workspace): cria o usuário via `supabase.auth.admin.inviteUserByEmail` com `redirectTo` para `/cliente/definir-senha`, registra em `client_users` (`workspace_id`, `client_id`, `user_id`, `email`, `invited_at`). O cliente recebe email, define senha em `/cliente/definir-senha` e é redirecionado para `/cliente`.

- [ ] **AC 3.8.2 (CRÍTICO)** — Isolamento por `client_id` com 403 explícito e testado:
  - RLS: novas policies `client_read` (SELECT) em `clients`, `ad_accounts`, `ad_campaigns`, `campaign_metrics` usando `auth_client_id()` (claim do JWT injetado pelo hook — nunca por parâmetro de request).
  - Camada de aplicação: o Route Handler `GET /api/cliente/metrics` valida que o `client_id` solicitado é igual ao claim do JWT; divergência retorna **HTTP 403 JSON** antes de qualquer query.
  - **Teste explícito obrigatório:** teste com sessão do cliente A solicitando dados do cliente B assere `403` (não apenas resultado vazio via RLS). RLS é a segunda camada, não a única evidência.

- [ ] **AC 3.8.3** — Anti-spoofing do claim: o `custom_access_token_hook` SEMPRE sobrescreve `user_metadata.client_id` — injeta o valor de `client_users` quando o usuário é cliente, e **remove o claim** (`null`) quando não é. Sem isso, qualquer usuário poderia forjar `user_metadata.client_id` via `updateUser()` e as policies `client_read` vazariam dados cross-workspace. Teste do hook cobre os dois caminhos.

- [ ] **AC 3.8.4** — Painel `/cliente` exibe: nome do cliente, métricas agregadas das contas de anúncio vinculadas ao seu `client_id` (investimento, resultados, CPL/ROAS conforme objetivo), filtro de período (7 / 14 / 30 dias) e indicador de saúde por cor por conta (verde/amarelo/vermelho — reutilizar a lógica de `apps/web/src/lib/dashboard/metrics.ts` da Story 3.7).

- [ ] **AC 3.8.5** — Cliente NÃO acessa área do gestor e vice-versa (middleware `proxy.ts`): usuário com claim `client_id` acessando rota do app do gestor → redirect `/cliente`; usuário gestor (sem claim) acessando `/cliente` → redirect `/dashboard`; visitante sem sessão em `/cliente` → redirect `/cliente/login`. Cliente não vê ações de pausar/ajustar, configurações de workspace nem dados de outros clientes — o painel é 100% visualização (nenhuma Server Action de escrita exposta em `/cliente`).

- [ ] **AC 3.8.6** — Sessão do cliente expira em 7 dias: middleware da área `/cliente` verifica `user.last_sign_in_at`; se anterior a 7 dias, executa signOut e redireciona para `/cliente/login`. Sem refresh automático prolongado além desse limite.

- [ ] **AC 3.8.7** — Testes unitários cobrindo: `inviteClientUser` (auth de gestor, client do workspace, IDOR), Route Handler `/api/cliente/metrics` (403 cross-client explícito — AC 3.8.2; 200 com claim correto; 401 sem sessão), lógica de expiração de 7 dias, hook (injeção e remoção do claim — pode ser teste SQL via migration de teste ou documentação de verificação manual se inviável em vitest).

## Dev Notes

### Migration 000018 — client_users + auth_client_id + hook + RLS

```sql
-- 1. Tabela de usuários-cliente
CREATE TABLE public.client_users (
  id           uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id uuid        NOT NULL REFERENCES public.workspaces(id) ON DELETE CASCADE,
  client_id    uuid        NOT NULL REFERENCES public.clients(id)    ON DELETE CASCADE,
  user_id      uuid        NOT NULL REFERENCES auth.users(id)        ON DELETE CASCADE,
  email        text        NOT NULL,
  invited_at   timestamptz NOT NULL DEFAULT now(),
  accepted_at  timestamptz,
  UNIQUE (user_id),                      -- 1 usuário = 1 cliente (modelo simples do MVP)
  UNIQUE (workspace_id, client_id, email)
);

ALTER TABLE public.client_users ENABLE ROW LEVEL SECURITY;

-- Gestor gerencia convites do próprio workspace
CREATE POLICY workspace_isolation ON public.client_users
  FOR ALL USING (workspace_id = auth_workspace_id());

GRANT SELECT, INSERT, UPDATE, DELETE ON public.client_users TO service_role;
GRANT SELECT, INSERT, UPDATE, DELETE ON public.client_users TO authenticated;

-- 2. auth_client_id() — espelho de auth_workspace_id() (000002)
CREATE OR REPLACE FUNCTION public.auth_client_id()
RETURNS uuid LANGUAGE sql STABLE AS $$
  SELECT (auth.jwt() -> 'user_metadata' ->> 'client_id')::uuid;
$$;

-- 3. Hook — CREATE OR REPLACE estendendo o de 000002.
--    ⚠️ CRÍTICO (AC 3.8.3): SEMPRE sobrescrever o claim client_id.
--    Se o usuário NÃO é client_user, o claim é REMOVIDO — sem isso,
--    user_metadata.client_id forjado via updateUser() vazaria dados
--    cross-workspace pelas policies client_read.
CREATE OR REPLACE FUNCTION public.custom_access_token_hook(event jsonb)
RETURNS jsonb
LANGUAGE plpgsql STABLE SECURITY DEFINER SET search_path = public AS $$
DECLARE
  claims            jsonb;
  user_workspace_id uuid;
  user_client_id    uuid;
BEGIN
  SELECT wm.workspace_id INTO user_workspace_id
  FROM   workspace_members wm
  WHERE  wm.user_id = (event ->> 'user_id')::uuid
  ORDER BY wm.created_at ASC LIMIT 1;

  SELECT cu.client_id INTO user_client_id
  FROM   client_users cu
  WHERE  cu.user_id = (event ->> 'user_id')::uuid
  LIMIT 1;

  claims := event -> 'claims';

  -- SEMPRE sobrescreve: valor real OU remoção (anti-spoofing).
  -- O strip do workspace_id fecha a mesma brecha da função original de 000002
  -- (que só setava quando encontrado, permitindo claim forjado via updateUser()
  -- persistir para usuários sem workspace). Como esta migration já reescreve a
  -- função inteira, o fix simétrico entra aqui — não é regressão, é correção.
  IF user_workspace_id IS NOT NULL THEN
    claims := jsonb_set(claims, '{user_metadata,workspace_id}', to_jsonb(user_workspace_id::text));
  ELSE
    claims := claims #- '{user_metadata,workspace_id}';
  END IF;

  IF user_client_id IS NOT NULL THEN
    claims := jsonb_set(claims, '{user_metadata,client_id}', to_jsonb(user_client_id::text));
  ELSE
    claims := claims #- '{user_metadata,client_id}';
  END IF;

  RETURN jsonb_set(event, '{claims}', claims);
END;
$$;
-- (GRANTs do hook para supabase_auth_admin já existem de 000002 — replace preserva)

-- 4. Policies client_read — SELECT only, aditivas às workspace_isolation (OR)
CREATE POLICY client_read ON public.clients
  FOR SELECT USING (id = auth_client_id());

CREATE POLICY client_read ON public.ad_accounts
  FOR SELECT USING (client_id = auth_client_id());

CREATE POLICY client_read ON public.ad_campaigns
  FOR SELECT USING (ad_account_id IN (
    SELECT id FROM public.ad_accounts WHERE client_id = auth_client_id()
  ));

CREATE POLICY client_read ON public.campaign_metrics
  FOR SELECT USING (ad_account_id IN (
    SELECT id FROM public.ad_accounts WHERE client_id = auth_client_id()
  ));

-- 5. Grants de leitura para authenticated (RLS restringe as linhas).
--    ad_accounts/ad_campaigns/campaign_metrics receberam grant apenas para
--    service_role em 000017 — a sessão do cliente precisa de SELECT.
GRANT SELECT ON public.ad_accounts      TO authenticated;
GRANT SELECT ON public.ad_campaigns     TO authenticated;
GRANT SELECT ON public.campaign_metrics TO authenticated;
```

### proxy.ts — blocos da área do cliente (T5 — consultar TD-002 antes)

Este arquivo já causou mais de um bug de ordem (Waves 3 e 4). O trecho abaixo mostra a **posição exata** dos dois blocos novos dentro da função `proxy()` atual, em relação aos blocos existentes. Diferente da `isServiceRoute` (que retorna ANTES da criação do client supabase), os blocos do cliente exigem a sessão — ficam APÓS `getUser()` e ANTES de `isAuthRoute`/`isDashboard`:

```typescript
export async function proxy(request: NextRequest) {
  const { pathname } = request.nextUrl

  // ── blocos existentes que retornam ANTES da sessão (NÃO mexer) ──────────
  const isPublicDashboard = pathname.startsWith('/dashboard/')
  if (isPublicDashboard) {
    return handleDashboardPasswordGate(request)
  }

  const isDashboardAuthApi = pathname.startsWith('/api/dashboard/')
  if (isDashboardAuthApi) {
    return NextResponse.next()
  }

  const isServiceRoute = /* ... Wave 4, inalterado ... */
  if (isServiceRoute) {
    return NextResponse.next()
  }

  let response = NextResponse.next({ request })
  const supabase = createServerClient(/* ... inalterado ... */)

  const { data: { user } } = await supabase.auth.getUser()

  // ══ NOVO BLOCO 1 — área do cliente (/cliente) — Story 3.8 ═══════════════
  // Posição: imediatamente após getUser(), ANTES de isAuthRoute/isDashboard.
  const clienteClaim = user?.user_metadata?.client_id as string | undefined
  const isClienteArea = pathname.startsWith('/cliente')

  if (isClienteArea) {
    const isClientePublic =
      pathname.startsWith('/cliente/login') || pathname.startsWith('/cliente/definir-senha')

    if (isClientePublic) return response
    if (!user) return NextResponse.redirect(new URL('/cliente/login', request.url))
    if (!clienteClaim) return NextResponse.redirect(new URL('/dashboard', request.url)) // gestor não pertence aqui

    // AC 3.8.6 — expiração de 7 dias (last_sign_in_at só muda em login real)
    const lastSignIn = user.last_sign_in_at ? new Date(user.last_sign_in_at).getTime() : 0
    if (Date.now() - lastSignIn > 7 * 24 * 60 * 60 * 1000) {
      await supabase.auth.signOut()
      return NextResponse.redirect(new URL('/cliente/login', request.url))
    }
    return response
  }

  // ══ NOVO BLOCO 2 — cliente autenticado fora da área dele → /cliente ═════
  // Posição: DEPOIS do bloco isClienteArea (que já retornou para /cliente/*)
  // e ANTES de isAuthRoute/isDashboard. Motivo: o cliente TEM `user`, então o
  // guard `isDashboard && !user` abaixo NÃO o barra — sem este bloco, um
  // cliente logado navegaria pelas páginas do gestor (AC 3.8.5). Cobre também
  // /login e / (cliente logado → sempre /cliente).
  if (clienteClaim) {
    return NextResponse.redirect(new URL('/cliente', request.url))
  }
  // ═════════════════════════════════════════════════════════════════════════

  const isAuthRoute = pathname.startsWith('/login') || pathname.startsWith('/register') || pathname.startsWith('/auth/')
  const isOnboarding = pathname === '/onboarding'
  const isDashboard = !isAuthRoute && !isOnboarding && !isPublicDashboard && pathname !== '/'

  const hasWorkspace = !!user?.user_metadata?.workspace_id

  // bloco existente — inalterado; cliente nunca chega aqui (retornou acima)
  if (isDashboard && !user) {
    const loginUrl = new URL('/login', request.url)
    loginUrl.searchParams.set('next', pathname)
    return NextResponse.redirect(loginUrl)
  }
  // ... demais blocos existentes inalterados ...
}
```

**Nota de segurança:** o claim em `user_metadata.client_id` só é confiável por causa do strip do hook (AC 3.8.3). O middleware usa o claim apenas para ROTEAMENTO; a autorização de dados vem do Route Handler (guard 403) + RLS.

### Route Handler — /api/cliente/metrics (AC 3.8.2)

```typescript
// GET /api/cliente/metrics?client_id=X&period=7|14|30
// 1. supabase.auth.getUser() — sem sessão → 401
// 2. claim = user.user_metadata.client_id — ausente → 403 (não é cliente)
// 3. searchParams.client_id !== claim → 403 JSON { error: 'Acesso negado' }  ← TESTE EXPLÍCITO
// 4. Query com o CLIENT DE SESSÃO (não service role!) — RLS client_read é a 2ª camada
// 5. Agregar com lib/dashboard/metrics.ts por período
```

O parâmetro `client_id` existe exatamente para tornar o 403 testável de forma explícita (AC do PRD). O guard compara com o claim ANTES de qualquer query. Nunca usar service role neste handler — a sessão do cliente + RLS é o modelo.

### Expiração de 7 dias — racional

`user.last_sign_in_at` só muda em login real (não em refresh de token). Middleware compara com `now - 7d`: sessão mais velha → signOut + redirect. Não exige config global de auth nem cookie adicional.

### Painel /cliente — reuso da 3.7

`apps/web/src/lib/dashboard/metrics.ts` (Story 3.7) já agrega métricas por conta/período para o dashboard público. Reutilizar as funções puras; a diferença é a fonte (sessão RLS em vez de service client) e o filtro de período interativo.

### O que NÃO existe neste painel (AC 3.8.5)

- Nenhuma Server Action de escrita importada em `/cliente/**`
- Nenhum link para rotas do app do gestor
- Nenhuma menção a outros clientes, workspace ou configurações
