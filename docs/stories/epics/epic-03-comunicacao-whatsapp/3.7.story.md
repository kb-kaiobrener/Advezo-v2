## Story

**As a** gestor de tráfego,
**I want to** gerar um link público de dashboard para cada cliente com meu logo e as métricas que eu selecionar,
**so that** eu substitua o envio de planilha ou print e o cliente possa ver os dados em tempo real sem precisar de login.

## Acceptance Criteria

- [ ] **AC 3.7.1** — UI em Clientes → [cliente] → Configurações: seção "Dashboard Compartilhável" com formulário contendo: seleção de métricas (multi-checkbox: Investimento, Impressões, Cliques, CTR, Conversões, Receita, ROAS, CPL), upload de logo (PNG/JPEG ≤ 2MB), proteção por senha (campo opcional — em branco = dashboard público). Botão "Gerar link" cria ou atualiza a config sem gerar novo token se já existe.

- [ ] **AC 3.7.2** — Link público no formato `/dashboard/[token]` onde `token` é um hex de 40 caracteres gerado server-side (`crypto.randomBytes(20).toString('hex')`). Token gerado uma única vez por cliente — reconfigurar métricas ou logo não muda o token. Server Action `saveDashboardConfig` faz upsert em `dashboard_configs` com conflito `workspace_id, client_id`.

- [ ] **AC 3.7.3** — Rota pública `/dashboard/[token]` implementada como Next.js Server Component. Fluxo server-side: (1) query `dashboard_configs` por token usando `createSupabaseServiceClient()` → (2) valida `is_active = true` → (3) se `password_hash != null`, redireciona para tela de senha → (4) agrega métricas do mês atual em `campaign_metrics` → (5) renderiza HTML. `SUPABASE_SERVICE_ROLE_KEY` nunca é enviada ao browser — conteúdo gerado exclusivamente no servidor (NFR-2).

- [ ] **AC 3.7.4** — Dashboard exibe: logo da agência (se configurado), nome do cliente, período exibido (mês atual), métricas selecionadas com valores formatados em PT-BR, data e hora da última atualização (campo `synced_at` de `campaign_metrics`).

- [ ] **AC 3.7.5** — Proteção por senha: quando `password_hash` configurado, rota pública exibe formulário de senha antes dos dados. Validação via POST para `/api/dashboard/[token]/auth` (Route Handler). Validação correta → seta cookie httpOnly `dash_auth_[token]` com valor HMAC-SHA256(token + password_hash + secret) — expira em 24h. Próximas visitas com cookie válido pulam a tela de senha sem nova validação de DB.

- [ ] **AC 3.7.6** — Link copiável na UI do gestor (botão "Copiar link" que executa `navigator.clipboard.writeText`). Botão "Desativar link" chama Server Action `deactivateDashboard(clientId)` que seta `is_active = false`. Dashboard desativado exibe página 404 (não retorna dados). Reativar com `saveDashboardConfig` gera nova config com **mesmo token** (não gera novo).

- [ ] **AC 3.7.7** — Upload de logo para Supabase Storage no bucket `dashboard-logos` (público, sem RLS — acesso via URL pública). Chave: `{workspaceId}/{clientId}/logo.{ext}`. Tamanho máximo 2MB validado no Server Action antes do upload. Logo URL armazenada em `dashboard_configs.logo_url`.

- [ ] **AC 3.7.8** — **Tempo de carregamento da rota pública abaixo de 2 segundos em 4G** (Lighthouse Performance Score ≥ 90, medido em produção Vercel). Requisitos técnicos: Server Component com `export const revalidate = 300` (cache 5 min no edge), sem JS desnecessário no bundle do cliente, `<Image>` do Next.js para o logo (otimização automática WebP + CDN), nenhum fetch client-side na rota pública, CSS inline via Tailwind (sem folha de estilo externa).

ALTER TABLE public.dashboard_configs ENABLE ROW LEVEL SECURITY;

CREATE POLICY workspace_isolation ON public.dashboard_configs
  FOR ALL USING (workspace_id = auth_workspace_id());
-- NOTA: rota pública /dashboard/[token] usa createSupabaseServiceClient() server-side
-- e não passa pelo RLS — o service client tem acesso irrestrito por design.

CREATE TRIGGER dashboard_configs_set_updated_at
  BEFORE UPDATE ON public.dashboard_configs

### Rota pública `/dashboard/[token]` (Server Component)

#### Regra de cache — decisão de design obrigatória

A decisão de cache depende da presença de senha e **deve ser tomada em runtime**, não no build:

| Caso | Diretiva | Motivo |
|------|----------|--------|
| `password_hash = null` (público) | `export const revalidate = 300` | Conteúdo idêntico para todos os visitantes — ISR de 5 min válido |
| `password_hash != null` (protegido) | `export const dynamic = 'force-dynamic'` | O que renderizar depende do cookie `dash_auth_[token]` enviado pelo visitante — não cacheável por definição |

**Não use `force-static` em nenhum caso:** força geração no build, incompatível com tokens dinâmicos. Não use `revalidate = 300` em dashboards com senha: o Next.js serviria a mesma resposta cacheada (sem PasswordGate) para todos os usuários.

**Implementação:** a rota detecta `config.password_hash` e exporta dinamicamente:

```typescript
// apps/web/src/app/dashboard/[token]/page.tsx
import { cookies } from 'next/headers'
import { notFound } from 'next/navigation'

// NÃO use export const dynamic nem export const revalidate no topo do arquivo.
// A diretiva de cache é definida per-request via generateStaticParams (não aplicável aqui)
// ou omitindo a diretiva (Next.js default: 'auto', que respeita o uso de cookies()).

export default async function DashboardPage({ params }: { params: { token: string } }) {
  const db = createSupabaseServiceClient()  // NUNCA exposta ao browser

  const { data: config } = await db
    .from('dashboard_configs')
    .select('*, clients(name)')
    .eq('token', params.token)
    .eq('is_active', true)
    .single()

  if (!config) notFound()

  // Dashboard com senha: verificar cookie por requisição (força dynamic rendering)
  if (config.password_hash) {
    const cookieStore = cookies()
    const authCookie = cookieStore.get(`dash_auth_${params.token}`)
    const secret = process.env.DASHBOARD_AUTH_SECRET!
    const expected = crypto.createHmac('sha256', secret)
      .update(params.token + config.password_hash)
      .digest('hex')
    const isAuthed = authCookie && crypto.timingSafeEqual(
      Buffer.from(authCookie.value),
      Buffer.from(expected)
    )
    if (!isAuthed) return <PasswordGate token={params.token} />
    // Ao usar cookies(), o Next.js automaticamente trata a rota como dynamic — sem cache.
  }
  // Dashboard público: sem cookies() → Next.js aplica ISR com revalidate padrão.
  // @dev: adicionar `export const revalidate = 300` SOMENTE se config.password_hash for null
  // — mas como não sabemos no build, a melhor estratégia é omitir a diretiva e deixar
  // o Next.js detectar automaticamente (dynamic = 'auto'): rotas sem cookies() recebem
  // ISR de acordo com o revalidate do layout pai ou sem cache se nenhum for definido.
  // Alternativa pragmática: definir `export const revalidate = 300` globalmente e aceitar
  // que dashboards com senha terão uma camada extra de cache na Vercel Edge (mitigado pelo
  // cookie httpOnly que o CDN não armazena — Vercel CDN respeita Vary: Cookie).

  // Busca métricas do mês atual
  // ...

  return <DashboardView config={config} metrics={...} />
}
```

**Por que não há risco de exposição da service key:**
- `createSupabaseServiceClient()` usa `SUPABASE_SERVICE_ROLE_KEY` — variável sem prefixo `NEXT_PUBLIC_`
- Server Components são executados apenas no servidor — nunca no bundle JS do cliente
- A rota não possui nenhum Client Component que faça fetch de dados com credenciais

**Variável `DASHBOARD_AUTH_SECRET`:** variável de ambiente dedicada para HMAC de senhas e cookies de autenticação do dashboard. Nunca reutilizar `SUPABASE_SERVICE_ROLE_KEY` para isso — segredos com propósitos distintos devem ser chaves distintas. Gerar uma vez com `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"` e adicionar ao `.env.local` e ao painel de variáveis do Vercel/Railway.

### Proteção por senha — Route Handler

```typescript
// apps/web/src/app/api/dashboard/[token]/auth/route.ts
import { NextRequest, NextResponse } from 'next/server'
import crypto from 'node:crypto'

export async function POST(req: NextRequest, { params }: { params: { token: string } }) {
  const { password } = await req.json() as { password?: string }
  if (!password) return NextResponse.json({ error: 'senha obrigatória' }, { status: 400 })

  const db = createSupabaseServiceClient()
  const { data: config } = await db
    .from('dashboard_configs')
    .select('password_hash, password_salt')
    .eq('token', params.token)
    .eq('is_active', true)
    .single()

  if (!config?.password_hash) return NextResponse.json({ error: 'token inválido' }, { status: 404 })

  const secret = process.env.DASHBOARD_AUTH_SECRET!
  const expected = crypto.createHmac('sha256', secret)
    .update(password + config.password_salt)
    .digest('hex')

  // timingSafeEqual previne timing attacks — NUNCA comparar hashes com !==
  const hashesMatch = crypto.timingSafeEqual(
    Buffer.from(expected),
    Buffer.from(config.password_hash)
  )
  if (!hashesMatch) {
    return NextResponse.json({ error: 'senha incorreta' }, { status: 401 })
  }

  // Cookie de auth — httpOnly, secure, SameSite=Lax, 24h
  const cookieVal = crypto.createHmac('sha256', secret)
    .update(params.token + config.password_hash)
    .digest('hex')

  const res = NextResponse.json({ ok: true })
  res.cookies.set(`dash_auth_${params.token}`, cookieVal, {
    httpOnly: true,
    secure: process.env.NODE_ENV === 'production',
    sameSite: 'lax',
    maxAge: 60 * 60 * 24,
    path: `/dashboard/${params.token}`,
  })
  return res
}
```

### Performance < 2s em 4G (AC 3.7.8)

Requisitos obrigatórios para atingir o AC:

| Requisito | Implementação |
|-----------|--------------|
| Cache de 5 min no edge | `export const revalidate = 300` |
| Sem JS no bundle do cliente | Rota `/dashboard/[token]` é Server Component puro; `PasswordGate` é o único Client Component |
| Logo otimizado | `<Image src={logo_url} ...>` do Next.js (WebP automático + CDN Vercel) |
| CSS inline | Tailwind (sem folha externa); sem Google Fonts na rota pública |
| Sem fetch client-side | Todos os dados carregados no Server Component |
| Agregação server-side | Uma query SUM agrupada em `campaign_metrics`, não N queries individuais |
