## Story

**As a** gestor de tráfego,
**I want to** configurar para cada cliente a frequência, o horário e o destinatário dos relatórios automáticos,
**so that** o envio ocorra sem intervenção minha no momento certo.

---

## Acceptance Criteria

- [ ] **AC 3.3.1** — UI em Clientes → [cliente] → Configurações: seção "Envio Automático de Relatório" com formulário de configuração contendo: frequência (diário / semanal / quinzenal / mensal), dia da semana (somente quando semanal ou quinzenal, dropdown Seg–Dom) ou dia do mês (somente quando mensal, número 1–28), hora do envio (HH:MM, padrão 09:00), tipo de destinatário (número individual / grupo de WhatsApp) e campo de ID do destinatário.

- [ ] **AC 3.3.2** — Destinatário suporta dois tipos (FR-B1):
  - **individual**: número E.164 (ex: `5511999998888`) — validado com regex `/^\d{10,15}$/` após remoção de `+` e espaços
  - **grupo**: JID Baileys no formato `XXXXXXXXXX@g.us` — validado com regex `/^\d+@g\.us$/`

- [ ] **AC 3.3.3** — Configuração salva em `report_schedules` por `(workspace_id, client_id)` — UNIQUE constraint (um schedule por cliente). Campos: `frequency`, `send_day`, `send_time`, `destination_type`, `destination_id`, `is_active`. Server Action `saveReportSchedule` faz upsert pelo conflito `workspace_id, client_id`.

- [ ] **AC 3.3.4** — Toggle "Ativo" por configuração: quando desativado (`is_active = false`), o cron da Story 3.5 ignora esse schedule. Toggle não apaga a configuração. Server Action `toggleReportSchedule(scheduleId, isActive)`.

- [ ] **AC 3.3.5** — Botão "Pré-visualizar relatório" na UI: chama Server Action `previewReport(clientId)`, que invoca `generateReport` com período dos últimos 30 dias e exibe o texto gerado em `<pre>` ou `<textarea readonly>` dentro de um modal. Nenhum envio real acontece.

- [ ] **AC 3.3.6** — Se o cliente não possui nenhum schedule salvo, UI exibe formulário em branco (estado "Não configurado"). Se já existe um schedule, UI carrega os valores atuais no formulário. Botão "Salvar" é o único ponto de persistência (sem auto-save).

## Dev Notes

### Migration 000013 — report_schedules

```sql
CREATE TABLE public.report_schedules (
  id               uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id     uuid        NOT NULL REFERENCES public.workspaces(id)  ON DELETE CASCADE,
  client_id        uuid        NOT NULL REFERENCES public.clients(id)     ON DELETE CASCADE,
  frequency        text        NOT NULL
                               CHECK (frequency IN ('daily', 'weekly', 'biweekly', 'monthly')),
  send_day         integer,    -- weekday 0-6 (weekly/biweekly) ou dia do mês 1-28 (monthly); NULL para daily
  send_time        time        NOT NULL DEFAULT '09:00:00',
  destination_type text        NOT NULL CHECK (destination_type IN ('individual', 'group')),
  destination_id   text        NOT NULL,  -- E.164 (individual) ou JID @g.us (group)
  is_active        boolean     NOT NULL DEFAULT true,
  created_at       timestamptz NOT NULL DEFAULT now(),
  updated_at       timestamptz NOT NULL DEFAULT now(),
  UNIQUE (workspace_id, client_id)
);

ALTER TABLE public.report_schedules ENABLE ROW LEVEL SECURITY;

CREATE POLICY workspace_isolation ON public.report_schedules
  FOR ALL USING (workspace_id = auth_workspace_id());

CREATE TRIGGER report_schedules_set_updated_at
  BEFORE UPDATE ON public.report_schedules
  FOR EACH ROW EXECUTE FUNCTION public.set_updated_at();

GRANT SELECT, INSERT, UPDATE, DELETE ON public.report_schedules TO service_role;
GRANT SELECT, INSERT, UPDATE, DELETE ON public.report_schedules TO authenticated;
```
