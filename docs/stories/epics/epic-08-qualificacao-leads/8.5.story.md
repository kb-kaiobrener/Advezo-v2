## Acceptance Criteria

- [x] **AC 8.5.1** — `GET /api/webhooks/meta/leadgen`: recebe parâmetros de challenge da Meta (`hub.mode`, `hub.verify_token`, `hub.challenge`). Faz lookup em `workspace_settings` por `meta_leadgen_verify_token` (usa o índice `ws_verify_token_idx`). Se `hub.mode = 'subscribe'` e token encontrado: responde com `hub.challenge` em texto puro (200). Se token não encontrado ou mode inválido: 403.

- [x] **AC 8.5.2** — `POST /api/webhooks/meta/leadgen`: **primeira operação** é validar a assinatura `X-Hub-Signature-256`. Calcula `HMAC-SHA256(rawBody, META_APP_SECRET)`, compara com o header (formato `sha256=<hex>`). Se inválida ou ausente: retorna **403** imediatamente, sem processar nada do payload. Esta validação não tem exceção — é obrigatória sem condição (FR-LA3).

- [x] **AC 8.5.3** — **[SEGURANÇA — AC CRÍTICO]** Testes explícitos de segurança da assinatura: (a) POST com `X-Hub-Signature-256` ausente → 403; (b) POST com assinatura com body adulterado → 403; (c) POST com assinatura correta → 200 (ACK). A assinatura deve ser calculada sobre o **raw body** (Buffer/string antes de parse JSON) — não sobre o objeto parseado.

- [x] **AC 8.5.4** — Após validação de assinatura bem-sucedida: extrair `meta_lead_id` e `ad_account_id` do payload Meta. Formato esperado: `{ object: 'page', entry: [{ changes: [{ value: { leadgen_id, ad_account_id, ... } }] }] }`. INSERT em `lead_processing_queue` com `status='pending'`. Violação de `lead_queue_meta_lead_id_unique` (23505) → ignorada silenciosamente (idempotente — Meta pode reenviar o mesmo evento).

- [x] **AC 8.5.5** — ACK imediato: `200 OK` retornado após validação da assinatura e enfileiramento. O response não espera o processamento do lead (é assíncrono via Story 8.6). Timeout Meta: < 20s; este endpoint deve responder em < 500ms (NFR-PERF-1).

- [x] **AC 8.5.6** — CORS **fechado** neste endpoint: sem `Access-Control-Allow-Origin`. Este é um webhook server-to-server (Meta → Advezo), não um formulário embed público — CORS aberto seria incorreto.

- [~] **AC 8.5.7** — Gestor configura `meta_leadgen_verify_token` por workspace nas integrações. Gerado automaticamente como string aleatória de 32 chars (`crypto.randomBytes(16).toString('hex')`) no momento da ativação da integração. Exibido uma única vez na UI para o gestor copiar e colar no Meta Business Manager.

## Dev Notes

### Validação de Assinatura — Raw Body Obrigatório

```typescript
// apps/web/src/app/api/webhooks/meta/leadgen/route.ts
import { createHmac, timingSafeEqual } from 'crypto'

export async function POST(request: Request) {
  // Capturar raw body ANTES de qualquer parse
  const rawBody = await request.text()

  const signature = request.headers.get('x-hub-signature-256')
  if (!signature) {
    return new Response('Missing signature', { status: 403 })
  }

  const expected = 'sha256=' + createHmac('sha256', process.env.META_APP_SECRET!)
    .update(rawBody)
    .digest('hex')

  // timingSafeEqual previne timing attacks
  const sigBuffer = Buffer.from(signature)
  const expBuffer = Buffer.from(expected)
  if (sigBuffer.length !== expBuffer.length || !timingSafeEqual(sigBuffer, expBuffer)) {
    return new Response('Invalid signature', { status: 403 })
  }

  const payload = JSON.parse(rawBody)
  // ... processar payload
}
```

**Crítico:** usar `timingSafeEqual` em vez de `===` para comparação de HMAC (previne timing attacks). Importar de `crypto` (nativo Node.js).

Extrair: `entry[0].changes[0].value.leadgen_id` (→ `meta_lead_id`) e `entry[0].changes[0].value.ad_account_id`. Um payload pode conter múltiplos `changes` — iterar sobre todos.

### Challenge Verification (FR-LA2)

```typescript
export async function GET(request: Request) {
  const { searchParams } = new URL(request.url)
  const mode = searchParams.get('hub.mode')
  const token = searchParams.get('hub.verify_token')
  const challenge = searchParams.get('hub.challenge')

  if (mode !== 'subscribe' || !token) {
    return new Response('Invalid verification', { status: 403 })
  }

  const supabase = createSupabaseServiceClient() // service role para bypass RLS
  const { data } = await supabase
    .from('workspace_settings')
    .select('workspace_id')
    .eq('meta_leadgen_verify_token', token)
    .single()

  if (!data) {
    return new Response('Token not found', { status: 403 })
  }

  return new Response(challenge, { status: 200 })
}
```

### INSERT em lead_processing_queue com Dedup

```typescript
// Iterar sobre todos os changes no payload
for (const entry of payload.entry ?? []) {
  for (const change of entry.changes ?? []) {
    if (change.field !== 'leadgen') continue
    const { leadgen_id, ad_account_id } = change.value

    // Buscar ad_account interno por external_account_id
    const { data: adAccount } = await supabase
      .from('ad_accounts')
      .select('id, workspace_id')
      .eq('external_account_id', ad_account_id)
      .single()

    if (!adAccount) continue // conta não mapeada — ignorar silenciosamente

    const { error } = await supabase.from('lead_processing_queue').insert({
      workspace_id: adAccount.workspace_id,
      meta_lead_id: leadgen_id,
      ad_account_id: adAccount.id,
      status: 'pending',
    })

    // 23505 = unique violation = webhook duplicado = ignorar
    if (error && error.code !== '23505') {
      console.error('[webhook-leadgen] Queue insert error:', error.message)
    }
  }
}

return new Response('OK', { status: 200 })
```

### Segurança

- `META_APP_SECRET`: nunca exposto em logs ou responses. Usado apenas para HMAC.
- Validação de assinatura com `timingSafeEqual` (anti-timing-attack).
- CORS fechado — este não é endpoint público para browsers.
- `meta_leadgen_verify_token` per workspace: gerado com `crypto.randomBytes(16).toString('hex')` — não derivado de nada previsível.

## Dev Agent Record (@dev — Dex)

### Decisões de implementação (não-óbvias)

1. **`request.text()` (não `arrayBuffer`) para raw body.** O HMAC é computado sobre a
   string exata; `text()` preserva os bytes assinados e mantém paridade com o snippet de
   referência da story. `request.json()` jamais é chamado antes da validação.

2. **`timingSafeEqual` com guarda de length.** `timingSafeEqual` LANÇA em buffers de
   tamanhos diferentes — então verificamos `sigBuffer.length !== expBuffer.length` antes,
   retornando 403 (e não deixando a exceção vazar). Comparação `===` foi rejeitada
   (vaza timing byte a byte).

3. **`[AUTO-DECISION]` `META_APP_SECRET` ausente → 500 (fail closed).** Secret não
   configurado nunca faz bypass da validação. (reason: um secret ausente deve falhar
   fechado, jamais abrir o webhook). Coberto por teste explícito.

4. **`[AUTO-DECISION]` Body assinado porém não-JSON → 400.** Caso a assinatura confira
   mas `JSON.parse` falhe, retornamos 400 (assinatura ok, mas payload corrompido). Não é
   403 — a autenticação passou. (reason: distinção semântica entre falha de auth e
   payload inválido).

5. **Lookup `ad_accounts.external_account_id`.** O `ad_account_id` do payload Meta é
   externo; mapeamos para o `id` interno (uuid) e `workspace_id` via
   `external_account_id` (coluna confirmada na migration 000004). Conta não mapeada →
   ignorada silenciosamente (não é erro, não bloqueia o ACK).

6. **Dedup global em `meta_lead_id`.** O índice `lead_queue_meta_lead_id_unique` é sobre
   `meta_lead_id` apenas (não composto com workspace). 23505 → idempotente/silencioso;
   o loop nunca retorna cedo, sempre devolvendo 200 (ACK) — mesmo em redelivery da Meta.

7. **CORS fechado (AC 8.5.6).** Nenhum header `Access-Control-Allow-*`, nenhum handler
   `OPTIONS` exportado. Webhook server-to-server não é chamado por browser. Difere
   conscientemente do endpoint público `leads/submit` (Story 8.3), que tem CORS aberto.

8. **Service-role (`createSupabaseServiceClient`).** Sem JWT, `auth_workspace_id()` é
   NULL e RLS bloquearia tudo — mesmo padrão dos crons (ARCH-1) e do submit público.
   Toda escrita é escopada por `workspace_id` derivado do `ad_accounts` mapeado.
