## Story

**I want to** que os leads de Lead Ads sejam processados automaticamente em até 90 segundos após a notificação da Meta,

## Dev Notes

### Workflow 2 — Processamento Assíncrono (SPEC.md Seção 7)

```
[cron Railway */1 * * * *]
POST /api/leads/process-queue (x-cron-secret)
  Para cada item pending (LIMIT 10, Promise.allSettled):
    1. GET /{meta_lead_id}?fields=field_data&access_token={token_conta} via Graph API
    2. Normaliza campos → phone, name, email (sempre disponível do Lead Ad)
    3. phone_hash = HMAC-SHA256(normalizePhone(phone), workspace_salt)
    4. email_encrypted = AES-256-GCM(email) [sempre para lead_ads]
    5. Encontra lead_ads_config por ad_account_id + leadgen_form_id
    6. INSERT leads { source='lead_ads', meta_lead_id, status='novo' }
       → 23505 leads_meta_lead_id_unique? → action:'none', skip CAPI
    7. Avaliar qualification_rules do client
    8. INSERT conversion_events 'Lead' → CAPI (user_data.lead_id=meta_lead_id, user_data.em=SHA256(email))
    9. Se qualificado: INSERT conversion_events 'CompleteRegistration' → CAPI
    10. UPDATE lead_processing_queue SET status='completed'
    [erro]: retry_count++; se >= 3 → status='failed'; INSERT sync_errors
```

### Estrutura do processItem

```typescript
async function processItem(item: LeadQueueItem): Promise<'completed' | 'none' | 'failed'> {
  const supabase = createSupabaseServiceClient() // service role — sem sessão de usuário

  // Marcar como 'processing'
  await supabase.from('lead_processing_queue')
    .update({ status: 'processing' })
    .eq('id', item.id)

  try {
    // Buscar ad_account e descriptografar token
    const { data: adAccount } = await supabase
      .from('ad_accounts')
      .select('id, workspace_id, encrypted_token, external_account_id')
      .eq('id', item.ad_account_id)
      .single()

    const token = decryptToken(adAccount.encrypted_token, process.env.TOKEN_ENCRYPTION_KEY!)

    // Chamar Graph API
    const res = await fetch(
      `https://graph.facebook.com/v19.0/${item.meta_lead_id}?fields=field_data,full_name,phone_number,email&access_token=${token}`
    )

    if (!res.ok) {
      const err = await res.json()
      throw new Error(err?.error?.message ?? 'Graph API error')
    }

    const leadData = await res.json()
    // ... normalizar, hash, criptografar, inserir
  } catch (err: any) {
    const newCount = item.retry_count + 1
    const failed = newCount >= 3

    await supabase.from('lead_processing_queue').update({
      status: failed ? 'failed' : 'pending',
      retry_count: newCount,
      last_error: err.message,
    }).eq('id', item.id)

    if (failed) {
      await supabase.from('sync_errors').insert({
        workspace_id: item.workspace_id,
        ad_account_id: item.ad_account_id,
        error_type: 'lead_processing_failed',
        error_message: err.message,
      })
    }
    return 'failed'
  }
}
```

### Dedup Explícito de Lead (AC 8.6.6)

```typescript
const { error: insertError } = await supabase.from('leads').insert({ ... })

if (insertError) {
  if (insertError.code === '23505') {
    // Double-delivery da Meta — lead já existe — marcar como completed sem CAPI
    await supabase.from('lead_processing_queue')
      .update({ status: 'completed', completed_at: new Date().toISOString() })
      .eq('id', item.id)
    return 'none' // action: 'none'
  }
  throw insertError
}
```

### Promise.allSettled — Falha Isolada

```typescript
const items = await fetchPendingItems(10) // LIMIT 10
const results = await Promise.allSettled(items.map(processItem))

let processed = 0, failed = 0, skipped = 0
for (const result of results) {
  if (result.status === 'fulfilled') {
    if (result.value === 'completed') processed++
    else if (result.value === 'none') skipped++
    else failed++
  } else {
    failed++
  }
}

return Response.json({ processed, failed, skipped })
```

### Nota sobre Supabase Client no Cron

Este endpoint é chamado pelo cron Railway **sem sessão de usuário** — sem JWT, sem cookie. Usar `createSupabaseServiceClient()` (SUPABASE_SERVICE_ROLE_KEY) para bypass de RLS. Padrão definido na Story 2.3 (Issue ARCH-1) — a factory `createSupabaseServiceClient` já deve existir em `@advezo/database` após a resolução do waiver ARCH-1. Se ainda não implementada: implementar aqui e exportar de `@advezo/database`.

### Segurança

- `encrypted_token` descriptografado em memória — nunca logado.
- `email_encrypted = AES-256-GCM(email)` armazenado; `SHA256(email)` calculado em memória apenas para CAPI (Story 8.7) — nunca persistido como hash.
- `phone_hash = HMAC-SHA256(normalizePhone(phone), workspace_salt)` — com salt, nunca SHA256 simples.
- Token Graph API vai como query param (exigência Meta/TLS) — não logado (mesmo padrão SEC-1 da Story 2.3).

### Decisões autônomas

- **[AUTO-DECISION 1]** Lookup de `lead_ads_config` apenas por `ad_account_id` + `workspace_id` → A fila (`lead_processing_queue`, Story 8.5) não persiste `leadgen_form_id` (só `meta_lead_id`/`ad_account_id`/`workspace_id`/`status`), então é impossível filtrar por `leadgen_form_id` como o AC 8.6.7 sugere. Config ausente → `client_id=null` (lead órfão, não é erro — AC 8.6.7 já prevê log de aviso).
- **[AUTO-DECISION 2]** `qualification_rules = []` para Lead Ads → `qualification_rules` é coluna de `lead_forms`; leads de Lead Ads não têm `lead_form` vinculado. Sem fonte de regras no schema → `rules=[]` → `evaluateQualificationRules` retorna `false` → dispara só `Lead`, sem `CompleteRegistration` automático. Semanticamente correto, sem inventar coluna.
- **[AUTO-DECISION 3]** Graph API `v19.0` (não v18.0) → consistência com todo o codebase (`capi/leads.ts`, `sync/meta.ts`, `alerts/meta-balance.ts`). Confirmado no Checkpoint 1.
- **[AUTO-DECISION 4]** Sem `next_retry_at`/`status='dead'` → não existem no schema aplicado (`status CHECK IN ('pending','processing','completed','failed')`). Segue AC 8.6.3 literal: `retry_count++` → `pending`; ao atingir 3 → `failed` + `sync_errors`.
