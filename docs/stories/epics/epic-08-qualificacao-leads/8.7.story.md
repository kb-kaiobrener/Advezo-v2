### Gate de Consentimento por Fonte — Código Crítico

```typescript
// apps/web/src/lib/capi/leads.ts

function buildUserData(lead: Lead): CAPIUserData {
  const userData: CAPIUserData = {
    // phone_hash já é HMAC-SHA256 — usar diretamente como 'ph'
    ph: [lead.phone_hash],
  }

  // Gate de email diferenciado por fonte (FR-CAPI3 e FR-CAPI4)
  if (lead.email_encrypted) {
    const includeEmail =
      lead.source === 'lead_ads' ||
      (lead.source === 'landing_page' && lead.consent_given_at !== null)

    if (includeEmail) {
      // Descriptografar em memória → SHA256 → descartar plaintext
      const emailPlaintext = decryptToken(lead.email_encrypted, process.env.TOKEN_ENCRYPTION_KEY!)
      const emailHash = createHash('sha256')
        .update(emailPlaintext.toLowerCase().trim())
        .digest('hex')
      userData.em = [emailHash]
      // emailPlaintext descartado — nunca gravado
    }
  }

  if (lead.source === 'lead_ads' && lead.meta_lead_id) {
    userData.lead_id = lead.meta_lead_id // sinal forte Meta
  }

  return userData
}
```

**CRÍTICO:** `SHA256(email)` calculado em memória, nunca persistido. `emailPlaintext` não aparece em logs. `email_encrypted` nunca enviado ao browser.

### Gate de Envio

```typescript
async function checkCAPIGate(
  workspaceId: string,
  adAccountId: string | null,
  supabase: SupabaseClient
): Promise<{ enabled: boolean; pixelId?: string; token?: string }> {
  const { data: ws } = await supabase
    .from('workspace_settings')
    .select('meta_conversions_api_enabled, meta_pixel_id')
    .eq('workspace_id', workspaceId)
    .single()

  if (!ws?.meta_conversions_api_enabled || !ws?.meta_pixel_id) {
    return { enabled: false }
  }

  if (!adAccountId) return { enabled: true, pixelId: ws.meta_pixel_id }

  const { data: acc } = await supabase
    .from('ad_accounts')
    .select('encrypted_token, status')
    .eq('id', adAccountId)
    .single()

  if (acc?.status === 'expired') return { enabled: false }

  const token = decryptToken(acc!.encrypted_token, process.env.TOKEN_ENCRYPTION_KEY!)
  return { enabled: true, pixelId: ws.meta_pixel_id, token }
}
```

### Registro em conversion_events (FR-CAPI7)

INSERT antes da chamada (status='pending'), UPDATE após:
- Sucesso (`events_received >= 1`): `status='sent'`
- Erro HTTP ou parsing: `status='failed'`, `error_message` da resposta Meta

Reenvio manual via botão na UI: padrão da Story 2.9 — reutiliza o mesmo mecanismo.

### Segurança

- `email_encrypted`: nunca logado, nunca enviado ao client, nunca exposto em payload de resposta.
- `SHA256(email)`: calculado em memória — descartado após uso. Não é persistido.
- `phone_hash` já é HMAC-SHA256 com salt — usado diretamente como `user_data.ph`.
- Token de acesso Meta: descriptografado em memória para chamada CAPI — não logado.
- `meta_pixel_id`: não é dado sensível em si, mas não deve aparecer em logs de debug.

### Decisões de Implementação (Dex / @dev — 2026-06-30)

**Gaps schema vs spec (AUTO-DECISIONS, elicitation override em modo interativo):**

1. **`workspace_settings.meta_conversions_api_enabled` NÃO existe.** O schema aplicado (migration 000000 + 000009) só tem `meta_pixel_id`. Decisão: o gate operativo é `meta_pixel_id IS NOT NULL` (CAPI está habilitada exatamente quando há pixel configurado) + `ad_accounts.status != 'expired'` + token decriptável. A flag booleana é lida defensivamente em `checkCAPIGate` — só bloqueia se a coluna existir e for `=== false`, honrando uma migration futura sem quebrar hoje. **Gap aberto:** migration futura deve adicionar a coluna booleana explícita se o produto exigir um toggle separado do pixel.

2. **`conversion_events` NÃO existe em nenhuma migration aplicada** (confirmado na própria migration 000009, que só estende o CHECK condicionalmente via `to_regclass`). Decisão: a persistência de auditoria (`skipped`/`pending`/`sent`/`failed`) é guardada por `conversionEventsExists()` (detecta erro `PGRST205`/`42P01`). Quando a tabela não existe, o resultado do disparo é retornado/logado mas não persistido (AC 8.7.4 funcionalmente pendente da migration de epic futuro — alinhado com AC 8.7.7 que confirma "nenhuma migration nesta story"). O núcleo está pronto: quando a tabela existir, a persistência ativa automaticamente.

3. **Token Meta vem de `ad_accounts.encrypted_token`**, não de uma coluna `meta_system_access_token` (que não existe). É o token que `checkCAPIGate` descriptografa com `decryptToken`.

4. **Conflito de assinatura com os callers da Story 8.3/8.4.** O núcleo canônico (`leads.ts`) opera sobre a linha completa de `leads` (`buildUserData(lead)`), como a story especifica. Os callers já mergeados (`/api/leads/submit/route.ts`, `app/actions/leads.ts`) chamam assinaturas legadas (`LeadCapiInput` objeto / `StatusCapiLead + leadId`). Decisão (sem regressão — Article V): `lead.ts` (singular) vira camada de compatibilidade que re-hidrata a linha de `leads` a partir do banco e delega ao núcleo. Os 204 testes existentes seguem verdes.

**Segurança (verificada):**
- `SHA256(email)` é calculado em `buildUserData` a partir de `decryptToken(email_encrypted)` em memória; o plaintext sai de escopo imediatamente. Teste dedicado prova que nem o hash nem o plaintext aparecem em nenhum `insert`/`update` de `conversion_events` — só dentro do body HTTP enviado à Meta.
- `phone_hash` usado diretamente como `ph` (já é HMAC-SHA256).
- Token Meta e `email_encrypted` nunca logados; mensagens de erro extraem só `error.message` da Meta (sem ecoar o body).

**Caminho `lead_ads` (dependência de 8.6):** implementado e testado por completo com leads mockados/fixtures de cada fonte. O teste de integração real ponta-a-ponta do caminho `source='lead_ads'` requer a Story 8.6 (process-queue) populando `leads` com `source='lead_ads'` — documentado como pendência de Wave 4. A lógica de gate (em sempre presente, `lead_id` setado) está coberta por testes unitários determinísticos.
