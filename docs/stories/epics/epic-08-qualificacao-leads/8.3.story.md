## Story

**As a** visitante de uma landing page de cliente,
**I want to** submeter meus dados através do formulário embed,
**so that** o gestor de tráfego receba meu lead qualificado e eu seja contatado.

## Acceptance Criteria

- [x] **AC 8.3.1** — `POST /api/leads/submit` (sem JWT — autenticado por `embed_token` no body) responde com CORS aberto (`Access-Control-Allow-Origin: *`, `Access-Control-Allow-Methods: POST, OPTIONS`, `Access-Control-Allow-Headers: Content-Type`). Preflight OPTIONS retorna 200.

- [x] **AC 8.3.2** — Validação do `embed_token`: lookup em `lead_forms` por `embed_token`. Se não encontrado → 401. Se `is_active = false` → 410 Gone. Toda operação de validação usa `embed_token` como autenticação — sem JWT, sem cookie.

- [x] **AC 8.3.3** — **[SEGURANÇA LGPD — AC CRÍTICO]** Server rejeita com 422 qualquer submissão que contenha `email` mas `consent = false` ou ausente — não é "aceitar e ignorar", é rejeição ativa (FR-LP3). Cenário de teste explícito obrigatório: payload `{ embed_token, name, phone, email: "x@x.com", consent: false }` → HTTP 422. Payload `{ embed_token, name, phone, email: "x@x.com" }` (consent ausente) → HTTP 422. Somente `consent: true` permite prosseguir.

- [x] **AC 8.3.4** — Rate limit por IP: máximo 5 submissões/hora por IP. Rate limit por embed_token: máximo 100 submissões/dia por embed_token. Implementado via query ao Supabase (count de leads por IP/token na janela de tempo — sem Redis). Excedido → 429 Too Many Requests.

- [x] **AC 8.3.5** — Processamento de dados para INSERT em `leads`: `phone_hash = HMAC-SHA256(normalizePhone(phone), workspace_salt)`, onde `workspace_salt` é recuperado do workspace do formulário. `email_encrypted = AES-256-GCM(email)` **somente se** `consent = true`. `consent_given_at = now()` **somente se** `consent = true`. Campos: `source = 'landing_page'`, `status = 'novo'`, `name`, `field_data` (campos customizados).

- [x] **AC 8.3.6** — Deduplicação por banco: violação do `UNIQUE INDEX leads_active_dedup (client_id, phone_hash) WHERE status NOT IN ('desqualificado')` → código PostgreSQL 23505 → resposta 409 `{ error: 'lead_already_exists' }` — comportamento idempotente.

- [x] **AC 8.3.7** — Após INSERT bem-sucedido (201), disparo assíncrono de evento CAPI `Lead` (não bloqueia response). Regras de email no payload CAPI: `user_data.em = SHA256(lowercase(trim(email)))` somente se `consent_given_at IS NOT NULL` (calculado em memória a partir do email decriptografado — nunca armazenado como hash). Response: `201 { lead_id }`.

- [x] **AC 8.3.8** — Validação Zod do body: `name` (string, required), `phone` (formato WhatsApp BR válido — `+55` seguido de DDD + 8 ou 9 dígitos), `embed_token` (string, required), `email` (string email, optional), `consent` (boolean, optional), `field_data` (object, optional). Campos inválidos → 422 com detalhes dos campos.

## Dev Notes

### Workflow 1 — Submissão LP (SPEC.md Seção 7)

```
Browser (domínio do cliente) → POST /api/leads/submit { embed_token, ... }

1. Valida embed_token → encontra lead_form → verifica is_active
2. Rate limit: IP + embed_token (Supabase query count)
3. Zod validation:
   - phone: formato WhatsApp BR válido
   - email presente? → consent obrigatório = true, senão 422 REJECTED
4. phone_hash = HMAC-SHA256(normalizePhone(phone), workspace_salt)
5. email_encrypted = AES-256-GCM(email) SE consent=true, senão NULL
6. INSERT leads { source='landing_page', status='novo', consent_given_at: now() se consent }
   → 23505? → 409 lead_already_exists (idempotente)
7. Avaliar qualification_rules → status = 'qualificado' se todas passam
8. INSERT conversion_events { event_name='Lead' } → POST Meta CAPI (assíncrono)
9. Se qualificado: INSERT conversion_events { event_name='CompleteRegistration' } → CAPI
10. 201 Created { lead_id }
```

### CORS (FR-LP7)

```typescript
const corsHeaders = {
  'Access-Control-Allow-Origin': '*',
  'Access-Control-Allow-Methods': 'POST, OPTIONS',
  'Access-Control-Allow-Headers': 'Content-Type',
}

export async function OPTIONS() {
  return new Response(null, { status: 200, headers: corsHeaders })

```

CORS aberto é correto para embed público. Segurança vem do `embed_token` (128 bits, não-adivinhável) + rate limit server-side.

### Gate de Consentimento — REJEIÇÃO ATIVA (FR-LP3)

```typescript
// Esta verificação deve ocorrer ANTES de qualquer processamento de dado
const hasEmail = !!body.email
const hasConsent = body.consent === true

if (hasEmail && !hasConsent) {
  return Response.json(
    { error: 'Consentimento obrigatório para compartilhamento de email (LGPD Art. 7º I)' },
    { status: 422, headers: corsHeaders }
  )
}
```

NUNCA "aceitar e ignorar o email" — o servidor deve rejeitar ativamente a submissão.

### phone_hash e email_encrypted

```typescript
import { createHmac, createCipheriv, randomBytes } from 'crypto'

function normalizePhone(phone: string): string {
  // Remove todos os não-dígitos, garante formato E.164 sem '+'
  return phone.replace(/\D/g, '')
}

function hashPhone(phone: string, salt: string): string {
  return createHmac('sha256', salt).update(normalizePhone(phone)).digest('hex')
}

// email_encrypted usa TOKEN_ENCRYPTION_KEY existente (AES-256-GCM)
// Formato: <iv_hex>:<authTag_hex>:<ciphertext_hex>
// Importar encryptToken de @advezo/utils (padrão já usado no Epic 2)
import { encryptToken, decryptToken } from '@advezo/utils'
```

`workspace_salt` recuperado de `workspaces.salt` (ou campo equivalente — verificar schema da Story 1.1). Se não existir campo salt no workspace, usar `workspace_id` como salt (documentar decisão).

### Segurança

- `email_encrypted`: nunca em texto plano em logs, responses, ou exemplos. Só AES-256-GCM armazenado.
- `SHA256(email)`: calculado em memória no momento do CAPI, nunca persistido como coluna.
- `phone_hash`: HMAC-SHA256 com `workspace_salt` — não SHA256 simples.
- `embed_token`: nunca logado.

### Decisões de implementação não-óbvias (IDS + AUTO-DECISIONS)

1. **REUSE `createSupabaseServiceClient()` (`@advezo/database`)** — Endpoint público não tem JWT, logo `auth_workspace_id()` é NULL e as policies RLS bloqueariam toda leitura/escrita. Usa-se service-role (ignora RLS), escopando TODA query por `workspace_id` explicitamente. Mesmo modelo de segurança dos crons (ARCH-1, Story 2.3). **Mitigação:** todas as 3 queries (lookup, rate limit IP, rate limit token) filtram por `leadForm.workspace_id`.

2. **[AUTO-DECISION] `workspace_salt` = `workspace_id`** — Não existe coluna `salt` em `workspaces` nem em `workspace_settings` (verificado por grep em todas as migrations). Conforme a própria story (linha 138), usa-se `workspace_id` como salt do HMAC-SHA256. `phone_hash = HMAC-SHA256(normalizePhone(phone), workspace_id)` — HMAC com chave, não SHA256 simples.

3. **[AUTO-DECISION] Rate limit de IP via `field_data->>_ip`** — Tabela `leads` não tem coluna `ip_address`. Conforme story (linha 155), o IP do cliente (de `x-forwarded-for` / `x-real-ip`) é gravado em `field_data._ip` (chave interna, não exibida na UI) e o count por IP filtra por `field_data->>_ip` na janela de 1h.

4. **REUSE `normalizePhone` + `encryptToken` (`@advezo/utils`)** — `normalizePhone` já entrega E.164 (`55`+DDD+9 dígitos); `encryptToken` é o AES-256-GCM padrão do Epic 2 (`TOKEN_ENCRYPTION_KEY`). `email_encrypted` e `consent_given_at` só são preenchidos quando `consent === true` && há email.

5. **CREATE stub CAPI (`lib/capi/lead.ts`)** — Story 8.7 ainda não existe. O stub define o contrato e isola a regra de segurança: `SHA256(email)` é calculado em memória em `buildLeadCapiUserData`, só com `consent_given_at !== null`, e NUNCA persistido como coluna. Disparo é fire-and-forget (`.catch(...)`), não bloqueia o 201.

6. **Ordem do gate de consentimento (AC 8.3.3)** — O gate roda como passo 3, imediatamente após o parse Zod e ANTES de: lookup de embed_token, hashing, criptografia e INSERT. Teste confirma `capture.payload === null` na rejeição (nenhum dado processado). Rejeição ativa (422 com mensagem LGPD), nunca "aceitar e ignorar".

### Observações / gaps (não-bloqueantes)

- **CodeRabbit não executado:** o ambiente atual não tem WSL instalado (`wsl.exe` indisponível), então o self-healing loop de CodeRabbit do dev-phase não rodou. Self-critique manual aplicado em seu lugar. Delegar varredura CodeRabbit ao @qa/@devops (que têm acesso WSL) no gate.
- **Dedup com `client_id` NULL:** o índice `leads_active_dedup (client_id, phone_hash)` não deduplicará leads quando `lead_forms.client_id` for NULL (NULLs são distintos em índice único do Postgres). Comportamento do schema (Story 8.1), não desta route. Na prática, AC 8.2.2 torna `client_id` obrigatório na criação do formulário, então o caminho normal sempre tem `client_id`.
- **`email` não-mockado nos testes:** `encryptToken`/`decryptToken` rodam de verdade — o teste de caminho feliz decripta `email_encrypted` e confere igualdade ao email original, provando AES-256-GCM real.
