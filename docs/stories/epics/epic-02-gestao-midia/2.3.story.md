### syncMetaAccount — Fluxo de dados

```typescript
// apps/web/src/lib/sync/meta.ts
import { decryptToken } from '@advezo/utils'
import { createSupabaseServerClient } from '@advezo/database'

export async function syncMetaAccount(adAccountId: string, workspaceId: string) {
  const supabase = createSupabaseServerClient()

  // 1. Buscar conta e descriptografar token
  const { data: account } = await supabase
    .from('ad_accounts')
    .select('encrypted_token, external_account_id')
    .eq('id', adAccountId)
    .single()

  const accessToken = decryptToken(account.encrypted_token, process.env.TOKEN_ENCRYPTION_KEY!)
  const actId = account.external_account_id  // formato: act_123456 ou 123456

  // 2. Chamar Meta Insights API
  // Fields: campaign_id, campaign_name, impressions, clicks, spend, actions, action_values
  // Attribution window: 7d_click (fixo — garante deduplicação consistente)
  const url = `https://graph.facebook.com/v19.0/${actId}/insights?` + new URLSearchParams({
    level: 'campaign',
    date_preset: 'last_7d',
    fields: 'campaign_id,campaign_name,impressions,clicks,spend,actions,action_values',
    'action_attribution_windows[]': '7d_click',
    access_token: accessToken,
  })

  const res = await fetch(url)

  if (!res.ok) {
    const errBody = await res.json()
    // OAuthException (#190) = token expirado
    const isExpired = errBody?.error?.code === 190
    await supabase.from('ad_accounts').update({
      status: isExpired ? 'expired' : 'error',
      error_message: errBody?.error?.message ?? 'Erro desconhecido',
    }).eq('id', adAccountId)
    await supabase.from('sync_errors').insert({
      workspace_id: workspaceId,
      ad_account_id: adAccountId,
      platform: 'meta',
      error_type: isExpired ? 'token_expired' : 'api_error',
      error_message: errBody?.error?.message ?? 'Erro desconhecido',
    })
    throw new Error(errBody?.error?.message)
  }

  const { data: insights } = await res.json()

  // 3. Upsert de campaigns + metrics
  for (const row of insights) {
    // Upsert campaign
    const { data: campaign } = await supabase.from('ad_campaigns').upsert({
      workspace_id: workspaceId,
      ad_account_id: adAccountId,
      platform: 'meta',
      external_campaign_id: row.campaign_id,
      name: row.campaign_name,
    }, { onConflict: 'ad_account_id,external_campaign_id' }).select('id').single()

    // Extrair conversões (action_type = offsite_conversion.fb_pixel_purchase ou purchase)
    const actions = row.actions ?? []
    const actionValues = row.action_values ?? []
    const purchaseTypes = ['purchase', 'offsite_conversion.fb_pixel_purchase']
    const conversions = actions
      .filter((a: any) => purchaseTypes.includes(a.action_type))
      .reduce((sum: number, a: any) => sum + Number(a.value), 0)
    const revenue = actionValues
      .filter((a: any) => purchaseTypes.includes(a.action_type))
      .reduce((sum: number, a: any) => sum + Number(a.value), 0)

    // Upsert métricas diárias (deduplicação por campaign_id + date)
    await supabase.from('campaign_metrics').upsert({
      campaign_id: campaign.id,
      workspace_id: workspaceId,
      date: row.date_start,
      impressions: Number(row.impressions ?? 0),
      clicks: Number(row.clicks ?? 0),
      spend: Number(row.spend ?? 0),
      conversions,
      revenue,
      synced_at: new Date().toISOString(),
    }, { onConflict: 'campaign_id,date' })
  }

  // 4. Atualizar last_synced_at e limpar status de erro anterior
  await supabase.from('ad_accounts').update({
    last_synced_at: new Date().toISOString(),
    status: 'active',
    error_message: null,
  }).eq('id', adAccountId)
}
```

### Issue ARCH-1 (HIGH) — Cron path falha sob RLS · **WAIVER CONDICIONAL**

- **severity:** high
- **category:** architecture / security
- **status:** WAIVED para merge · **BLOCKER para o primeiro deploy de produção do cron**

**Diagnóstico (confirmado por leitura de código, não inferido):**
`createSupabaseServerClient` (`packages/database/src/server.ts`) usa `NEXT_PUBLIC_SUPABASE_ANON_KEY` e lê cookies via `next/headers`. No contexto do cron Railway não há cookies → o JWT é anônimo → `auth_workspace_id()` retorna NULL → toda leitura/escrita RLS-scoped retorna zero linhas **silenciosamente**. O `SELECT` de `ad_accounts` no `route.ts:39` viria vazio e o loop não sincronizaria nada — sem erro, sem log. Pior que uma exceção: é uma falha silenciosa, que viola o espírito do NFR-4 no caminho mais crítico (o automático).

**Por que CONCERNS e não FAIL:** A regra de decisão do gate é binária — *se já existe mecanismo de service client, FAIL (fix trivial); se não existe, CONCERNS (roadmap de fix)*. Verifiquei: `@advezo/database` exporta **apenas** `createSupabaseServerClient` + `createSupabaseBrowserClient` (`index.ts`). NÃO há `createSupabaseServiceClient`. O único uso de service-role no repo é um `createClient(...)` cru dentro de `__tests__/integration/auth-workspace-id.test.ts` — helper de teste, não factory de produção. Além disso, `syncMetaAccount` instancia o client **internamente** (`meta.ts:110`, rígido), não o recebe por parâmetro. O fix é net-new arquitetura em 3 arquivos + novo export — fora dos ACs (AC 2.3.2 fixou `syncMetaAccount` no client de sessão). Logo: CONCERNS com roadmap, não FAIL.

**Roadmap de fix obrigatório (antes de habilitar o cron em prod) — story de follow-up:**

- **severity:** informational
