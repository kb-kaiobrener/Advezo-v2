## Story

**As a** gestor de tráfego,
**I want to** receber alertas quando o saldo de uma conta de anúncio está prestes a se esgotar,
**so that** eu possa agir antes que as campanhas parem de veicular por falta de verba — sem precisar monitorar manualmente.

## Acceptance Criteria

- [x] **AC 2.9.1** — Cron diário (Railway, `0 7 * * *` — após o sync das 06:00) verifica saldo projetado de cada conta ativa. Lógica: `avg_daily_spend` = média de spend dos últimos 7 dias; `projected_days` = budget_disponível / avg_daily_spend. Alertar quando `projected_days < threshold` (threshold a definir no checkpoint).

- [x] **AC 2.9.2** — **Sem alertas duplicados**: antes de inserir novo alerta, verificar se existe alerta ativo (`resolved_at IS NULL`) do mesmo tipo para a mesma conta dentro da janela de deduplicação (a definir no checkpoint). Se existir, não inserir.

- [x] **AC 2.9.3** — **Falha no cron de detecção nunca silenciosa (NFR-4)**: se a detecção falhar para uma conta (ex: erro de cálculo, query falha), registrar em `sync_errors` com `error_type='alert_detection_failed'`. Outras contas continuam sendo processadas.

- [x] **AC 2.9.4** — Badge de alerta visível na UI: conta com alerta ativo exibe indicador visual em `AdAccountCard.tsx` e/ou na listagem de campanhas. Badge removido quando `alerts.resolved_at` é preenchido.

- [x] **AC 2.9.5** — Alertas visíveis na página `/settings/integrations` por conta: lista de alertas ativos com tipo, criado_em e ação "Marcar como resolvido" (manual — preenche `resolved_at`).

- [x] **AC 2.9.6** — Alerta marcado como resolvido automaticamente quando: (a) o cron roda novamente e `projected_days >= threshold * 2` (folga de 2x para evitar flicker), OU (b) usuário marca manualmente.

- [x] **AC 2.9.7** — Tabela `alerts` com RLS por workspace. `alert_type` inicial: `'low_balance'`. Extensível para outros tipos (Epic 3 pode adicionar `'whatsapp_disconnected'` etc.).

- [x] **AC 2.9.8** — Testes unitários: (a) detecção correta quando projected_days < threshold; (b) sem detecção quando projected_days >= threshold; (c) deduplicação: segunda execução com alerta já ativo não insere duplicata; (d) falha de cálculo → registrada em sync_errors, não propaga exceção.

- [x] **AC 2.9.9** — Dispatch via WhatsApp **fora do escopo desta story** — apenas detecção local e UI. O Epic 3 (Story 3.6) consome a tabela `alerts` para envio.

## Dev Notes

### Schema proposto (a confirmar no checkpoint)

```sql
CREATE TABLE alerts (
  id            uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id  uuid        NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE,
  ad_account_id uuid        NOT NULL REFERENCES ad_accounts(id) ON DELETE CASCADE,
  alert_type    text        NOT NULL CHECK (alert_type IN ('low_balance')),
  threshold_days int        NOT NULL,
  projected_days numeric    NOT NULL,
  created_at    timestamptz NOT NULL DEFAULT now(),
  resolved_at   timestamptz
);
```

### Cálculo de saldo projetado

```typescript
// avg_daily_spend: média de spend dos últimos 7 dias (de campaign_metrics)
// available_budget: a definir — daily_budget * dias_restantes OU lifetime_budget?
// → DECISÃO PENDENTE DE CHECKPOINT

export function calculateProjectedDays(avgDailySpend: number, availableBudget: number): number {
  if (avgDailySpend === 0) return Infinity // conta sem gasto → sem risco
  return availableBudget / avgDailySpend
}
```

### Deduplicação (estratégia a confirmar)

```typescript
// Antes de inserir alerta, verificar existência:
const { data: existing } = await supabase
  .from('alerts')
  .select('id')
  .eq('ad_account_id', accountId)
  .eq('alert_type', 'low_balance')
  .is('resolved_at', null)
  .single()

if (existing) return // já existe alerta ativo — não duplicar
```

### Completion Notes

- **CP4 (dedup no DB):** A invariante "no máximo 1 alerta ativo por (conta, tipo)" é garantida pelo índice único parcial `alerts_active_unique ... WHERE resolved_at IS NULL`. O código tenta o INSERT e captura `code 23505` (unique_violation) tratando como "já existe" — sem propagar exceção e sem duplicar. Testado explicitamente (alerts-detect.test.ts cenário 3): a query prévia retorna "sem alerta ativo" mas o INSERT é rejeitado pela constraint, provando que a dedup é do banco e não depende só da checagem prévia.
- **CP5 (saldo via API real):** Meta `?fields=balance` (centavos → BRL); Google `account_budget` GAQL (`approved - served` micros → BRL). Nenhum proxy via `daily_budget` local. Tokens descriptografados em memória; nunca em logs nem em mensagens de erro (teste verifica que a mensagem não contém o token).
- **Resiliência (AC 2.9.3 + requisito QA #2):** `detectAccountBalance` jamais lança — toda falha (token ausente, fetch de rede, leitura de métricas) vira `sync_errors` com `error_type='alert_detection_failed'` e retorno `ok:false`. O route ainda envolve cada chamada em try/catch (defesa em profundidade). Teste com 3 contas onde a do meio falha o fetch confirma que a 1ª e a 3ª são processadas e só a 2ª registra falha.
- **avg_daily_spend por conta:** a média de gasto é escopada por conta via embed `campaign_metrics → ad_campaigns!inner(ad_account_id)` com `.eq('ad_campaigns.ad_account_id', account.id)` — a projeção reflete o ritmo de gasto da conta, não do workspace. `averageDailySpend` divide pelo número de DIAS da janela (7), não pelo número de linhas, então dias sem gasto contam como 0.
- **Auto-resolve (AC 2.9.6a):** folga de 2x (14 dias) evita flicker. Ordem no detect: se há alerta ativo e projeção ≥ 14 → resolve; senão, se não há alerta ativo e projeção < 7 → cria.
- **[AUTO-DECISION]** Extração de `detectAccountBalance` para `lib/alerts/detect.ts` em vez de inline no route → decisão: separar lógica de orquestração da camada HTTP, espelhando o padrão `route → lib/sync/*` das Stories 2.3/2.4 (testabilidade unitária do loop sem subir o handler).
- **[AUTO-DECISION]** Threshold projetado por conta usando `maybeSingle()` na busca de alerta ativo (vs `.single()`) → evita erro quando não há linha (single lança em 0 rows; maybeSingle devolve null).
- **CodeRabbit self-healing:** não executado — WSL não está instalado neste ambiente (`wsl.exe --install` ausente). Conforme error-handling documentado em `dev-develop-story.md`, o self-healing é pulado com revisão manual. Substituído localmente por ESLint + tsc + suíte completa (128 testes). Recomenda-se rodar CodeRabbit no ambiente do @devops antes do push.
- **Validações finais:** `tsc --noEmit` ✅ (apps/web e packages/types), `eslint src` ✅ nos arquivos da story, `vitest run` ✅ 128 passed / 4 skipped / 20 todo — 0 regressões sobre o baseline de 92.
