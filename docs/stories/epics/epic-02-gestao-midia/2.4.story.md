## Dev Notes

### syncGoogleAccount — Fluxo de dados

```typescript
// apps/web/src/lib/sync/google.ts
import { decryptToken } from '@advezo/utils'
import { refreshGoogleToken } from '@/lib/oauth/google'
import { createSupabaseServerClient } from '@advezo/database'

export async function syncGoogleAccount(adAccountId: string, workspaceId: string) {
  const supabase = createSupabaseServerClient()

  const { data: account } = await supabase
    .from('ad_accounts')
    .select('encrypted_token, encrypted_refresh_token, external_account_id')
    .eq('id', adAccountId)
    .single()

  const customerId = account.external_account_id  // Google customer ID (sem hífens)
  let accessToken = decryptToken(account.encrypted_token, process.env.TOKEN_ENCRYPTION_KEY!)

  const doQuery = async (token: string) => fetch(
    `https://googleads.googleapis.com/v17/customers/${customerId}/googleAds:search`,
    {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${token}`,
        'developer-token': process.env.GOOGLE_ADS_DEVELOPER_TOKEN!,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({
        query: `SELECT campaign.id, campaign.name, campaign.status,
                       metrics.impressions, metrics.clicks, metrics.cost_micros,
                       metrics.conversions, metrics.conversions_value
                FROM campaign
                WHERE segments.date DURING LAST_7_DAYS`,
      }),
    }
  )

  let res = await doQuery(accessToken)

  // Auto-refresh em UNAUTHENTICATED
  if (res.status === 401) {
    try {
      accessToken = await refreshGoogleToken(
        adAccountId,
        account.encrypted_refresh_token!,
        process.env.TOKEN_ENCRYPTION_KEY!
      )
      res = await doQuery(accessToken)
    } catch (refreshErr) {
      // Refresh falhou → token revogado
      await supabase.from('sync_errors').insert({
        workspace_id: workspaceId,
        ad_account_id: adAccountId,
        platform: 'google',
        error_type: 'refresh_token_invalid',
        error_message: (refreshErr as Error).message,
      })
      await supabase.from('ad_accounts').update({
        status: 'error',
        error_message: 'Refresh token inválido — reconectar conta',
      }).eq('id', adAccountId)
      throw refreshErr
    }
  }

  if (!res.ok) {
    const errBody = await res.json()
    const msg = errBody?.error?.message ?? 'Google Ads API error'
    await supabase.from('sync_errors').insert({
      workspace_id: workspaceId,
      ad_account_id: adAccountId,
      platform: 'google',
      error_type: 'api_error',
      error_message: msg,
    })
    await supabase.from('ad_accounts').update({ status: 'error', error_message: msg }).eq('id', adAccountId)
    throw new Error(msg)
  }

  const { results } = await res.json()

  for (const row of results ?? []) {
    const c = row.campaign
    const m = row.metrics

    const { data: campaign } = await supabase.from('ad_campaigns').upsert({
      workspace_id: workspaceId,
      ad_account_id: adAccountId,
      platform: 'google',
      external_campaign_id: c.id,
      name: c.name,
      status: c.status?.toLowerCase(),
    }, { onConflict: 'ad_account_id,external_campaign_id' }).select('id').single()

    await supabase.from('campaign_metrics').upsert({
      campaign_id: campaign.id,
      workspace_id: workspaceId,
      date: new Date().toISOString().split('T')[0],  // data de referência do sync
      impressions: Number(m.impressions ?? 0),
      clicks: Number(m.clicks ?? 0),
      spend: (Number(m.cost_micros ?? 0) / 1_000_000),
      conversions: Number(m.conversions ?? 0),
      revenue: Number(m.conversions_value ?? 0),
      synced_at: new Date().toISOString(),
    }, { onConflict: 'campaign_id,date' })
  }

  await supabase.from('ad_accounts').update({
    last_synced_at: new Date().toISOString(),
    status: 'active',
    error_message: null,
  }).eq('id', adAccountId)
}
```

### Nota sobre GAQL e datas

O GAQL com `DURING LAST_7_DAYS` retorna dados segmentados por dia (7 linhas por campanha). Para cada linha, usar `segments.date` (incluir no SELECT) para popular `campaign_metrics.date`. Ajustar o query acima para incluir `segments.date` no SELECT e usar `row.segments.date` no upsert de campaign_metrics.

## Dev Agent Record

### ARCH-1 — Service Client para cron endpoints (RESOLVIDO)

O Quality Gate da Story 2.3 documentou que os crons (`/api/sync/meta`, `/api/sync/google`) usavam client cookie-based: sem JWT no cron Railway, `auth_workspace_id()` é NULL e as policies RLS bloqueiam as escritas silenciosamente. Fix entregue nesta story:
- **Passo A:** `createSupabaseServiceClient()` (service-role, ignora RLS) em `@advezo/database`, com export `./service`. URL via `SUPABASE_URL` com fallback para `NEXT_PUBLIC_SUPABASE_URL` (URL é pública; só a service-role key é secreta). NUNCA exposto em Client Components nem via `NEXT_PUBLIC_*`.
- **Passo B:** `syncMetaAccount` e `syncGoogleAccount` aceitam `supabaseClient?` opcional (injeção de dependência — testável).
- **Passo C/D:** ambos os crons usam o service client para listar contas E o injetam no sync.

### Desvios das Dev Notes (correções aplicadas)

1. **`refreshGoogleToken` retorna o token CRIPTOGRAFADO** (não o access_token puro — comportamento real da Story 2.2). O sync descriptografa o retorno antes de usá-lo como Bearer no retry.
2. **Mapeamento de status Google** — `ENABLED/PAUSED/REMOVED/UNKNOWN` → valores do CHECK de `ad_campaigns` (`active/paused/deleted/archived`). O `c.status?.toLowerCase()` sugerido nas Dev Notes produziria `enabled`/`removed`, violando o CHECK constraint.
3. **`segments.date` incluído no SELECT** e usado como `campaign_metrics.date` (uma linha por campanha por dia), conforme a nota da própria story.

### Limitação conhecida (para o Quality Gate)

`refreshGoogleToken` (Story 2.2) persiste o novo `encrypted_token` via seu PRÓPRIO client cookie-based interno. No caminho de cron essa escrita interna é bloqueada por RLS, mas a função **retorna** o token novo, que o sync usa em memória para o retry — o sync corrente conclui com sucesso. O token será re-persistido no próximo sync com sessão (Server Action). Não bloqueia a funcionalidade; um follow-up opcional seria parametrizar `refreshGoogleToken` para aceitar o service client (fora do escopo desta story para não alterar a interface/tests da 2.2).

### Decisão sobre a limitação do `refreshGoogleToken` (Critério 10) — CONCERNS, não FAIL

**Análise do fluxo completo de recuperação no path de cron:**

1. Token armazenado expirado → `doQuery` retorna 401.
2. `refreshGoogleToken(adAccountId, encrypted_refresh_token, key)` usa o **refresh_token** (longa duração) para obter novo access_token, re-criptografa e **tenta** persistir via seu client cookie-based interno. No cron (sem JWT) essa UPDATE é RLS-blocked silenciosamente — **mas a função retorna** o `newEncryptedToken`.
3. `syncGoogleAccount` descriptografa o retorno e refaz `doQuery` → **sucesso**. Campaigns/metrics e `last_synced_at` são gravados via o **service client injetado** (não RLS-blocked).

**Por que NÃO é FAIL (não há ciclo de falha 401):**
- O critério de FAIL era: token desatualizado no banco faz o **próximo** cron também falhar em ciclo. Isso **não** ocorre.
- No próximo cron run, o `encrypted_token` ainda está velho → 401 → `refreshGoogleToken` é chamado de novo com o **`encrypted_refresh_token`**, que **nunca é sobrescrito** pelo sync (grep confirmou: escrito só no callback OAuth da 2.2; lido — nunca atualizado — pelo sync). O refresh_token Google é longa-duração e não rotacionado aqui → o refresh **sempre tem sucesso** → retry tem sucesso.
- Resultado: **self-healing a cada run**. O único custo é 1 chamada HTTP de refresh extra por cron run, até um sync manual com sessão re-persistir o access_token. Sem falha de sync, sem ciclo de 401.

**Veredito do critério 10:** **CONCERNS** — comportamento funcionalmente correto e estável; ineficiência marginal (um refresh redundante por run no path 100%-cron). Aceitável para Wave 2.
