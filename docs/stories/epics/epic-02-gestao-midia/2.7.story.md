## Acceptance Criteria

- [x] **AC 2.7.1** — Botões "Pausar" e "Ativar" disponíveis em `CampaignRow` para campanhas `active` e `paused` respectivamente. Somente um estado é visível por vez — campanha ativa mostra "Pausar", campanha pausada mostra "Ativar".

- [x] **AC 2.7.2** — Antes de chamar a API externa, registrar em `action_log`: `user_id`, `workspace_id`, `ad_account_id`, `campaign_id` (external), `action_type`, `old_value`, `new_value`, `status='pending'`. Após resposta da API, atualizar `status` para `'success'` ou `'failed'` com `api_error`.

- [x] **AC 2.7.3** — **Se a API externa retornar erro** (qualquer status ≠ 2xx, ou payload de erro): `ad_campaigns.status` e `ad_campaigns.daily_budget` NÃO são atualizados. O estado local permanece inalterado. `action_log.status` → `'failed'`, `action_log.api_error` → mensagem da plataforma. UI exibe o erro inline.

- [x] **AC 2.7.4** — **Se a API retornar sucesso**: `ad_campaigns.status` e/ou `ad_campaigns.daily_budget` são atualizados para refletir a mudança confirmada. `action_log.status` → `'success'`. UI reflete o novo estado.

- [x] **AC 2.7.5** — Campo de ajuste de orçamento diário inline em `CampaignRow`: input numérico editável, confirmado com botão "Salvar" (não salva automaticamente ao perder foco). Valor mínimo: plataforma decide — erros de validação da API são exibidos inline.

- [x] **AC 2.7.6** — Todas as ações verificam que a campanha (`ad_campaign_id`) pertence ao workspace autenticado antes de qualquer operação. RLS já garante isso na leitura; a Server Action deve confirmar também na escrita (`ad_account_id` pertence ao workspace).

- [x] **AC 2.7.7** — `action_log` tem RLS: usuário autenticado vê apenas logs do próprio workspace. Administrador do workspace pode ver logs de todos os membros.

- [x] **AC 2.7.8** — Erros de rede ou timeout para a API externa são tratados como falha: `action_log.status='failed'`, `api_error='timeout'` ou mensagem de rede. NÃO atualizar estado local.

- [x] **AC 2.7.9** — Testes unitários: (a) sucesso → status atualizado, action_log success; (b) falha API → status local preservado, action_log failed; (c) ownership check rejeita campanha de outro workspace.

- [x] **AC 2.7.10 (integração)** — Teste de integração com sandbox Meta Test Ad Account e Google Ads Test Customer para pausar/ativar real. Skipped sem credenciais (padrão das Waves 1-2).

## Dev Notes

### Fluxo de ação (decisão pendente de checkpoint)

```
1. auth guard (getAuthenticatedWorkspace)
2. verificar ownership: ad_campaign pertence ao workspace
3. INSERT action_log (status='pending')
4. chamar API externa (Meta ou Google)
5a. sucesso → UPDATE ad_campaigns + UPDATE action_log status='success'
5b. falha → UPDATE action_log status='failed', api_error=msg; NÃO tocar ad_campaigns
```

### Meta API — Pause/Activate
```
PATCH https://graph.facebook.com/v19.0/{campaign_id}
  ?access_token={decrypted_token}
  &status=PAUSED|ACTIVE
```

### Meta API — Update Budget
```
PATCH https://graph.facebook.com/v19.0/{campaign_id}
  ?access_token={decrypted_token}
  &daily_budget={value_in_cents}
```

### Google Ads API — Pause/Activate
```
POST https://googleads.googleapis.com/v16/customers/{customer_id}/campaigns:mutate
  Authorization: Bearer {access_token}
  Body: { operations: [{ update: { resourceName, status: PAUSED|ENABLED }, updateMask: "status" }] }
```

## Dev Notes — Implementação (Dex / @dev)

### Decisões de alinhamento com o codebase
- **Google Ads API v17** (não v16): alinhado a `lib/sync/google.ts` e `lib/oauth/google.ts`, que já usam v17. O spec mencionava "mesma version que syncGoogleAccount usa" — essa é v17.
- **Meta Graph v19.0**: alinhado a `lib/sync/meta.ts` (mesma constante GRAPH_VERSION).
- **Token na ad_accounts, não em ad_campaigns**: a query da Server Action faz join `ad_accounts!inner(...)` para obter `encrypted_token`, `encrypted_refresh_token`, `external_account_id`. `ad_campaigns` não tem token.
- **`daily_budget` (numeric, BRL)**: orçamento editável é `daily_budget`. Meta espera centavos (`*100`); Google espera micros (`*1_000_000`) — conversão feita nas funções de plataforma.
- **AC 2.7.7 (RLS admin)**: a policy `workspace_isolation USING (workspace_id = auth_workspace_id())` já satisfaz o requisito — como o JWT injeta o workspace_id do membro, todos os membros (incluindo admin) veem os logs do próprio workspace. Não há necessidade de policy por role separada.

### Pontos de atenção para QA (@architect)
- **Refresh-on-401 Google**: implementado via callback `refreshGoogleToken` injetado pela Server Action; o token renovado é usado em memória para o retry e re-persistido pelo próprio refreshGoogleToken.
- **Orçamento Google (resolvido)**: o orçamento canônico do Google Ads vive no recurso `CampaignBudget` (resourceName próprio). `googleUpdateBudget` agora busca esse resourceName via GAQL (`fetchCampaignBudgetResourceName` — `SELECT campaign.campaign_budget ...`) ANTES de qualquer mutação; se não encontrar (ou erro de API/rede) retorna falha sem mutar (estado local preservado). A mutação usa o endpoint próprio `campaignBudgets:mutate` (distinto de `campaigns:mutate`) com `updateMask: 'amount_micros'`. Refresh-on-401 mantido. Não há mais tech debt aqui.
- **CodeRabbit self-healing NÃO executado**: WSL não está instalado neste ambiente (`wsl --install` necessário). O QA Gate foi validado via TypeCheck + ESLint (0 erros/0 warnings) + suíte completa de testes (89 passed, 6 skipped, 15 todo). Recomenda-se rodar CodeRabbit no ambiente do @devops antes do push.
